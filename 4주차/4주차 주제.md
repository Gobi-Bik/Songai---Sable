* #### 암호화·인코딩·해시의 차이

  1. 개념

     1. 암호화 : 평문을 암호문으로 바꿔서 허가되지 않은 사람이 내용을 보지 못하게 하는 과정

        * 키가 존재
     2. 인코딩 : 데이터를 다른 시스템이나 형식에서 다루기 쉽게 표현 방식만 바꾸는 것

        * 키 필요 없음
     3. 해시 : 고정된 길이의 값으로 바꾸는 일방향 변환

        * 되돌릴 수 없음
  2. 필요 이유

     * 보안 목적이 각각 다름

       1. 암호화 : 내용을 나중에 다시 읽어야 한다
       2. 인코딩 : 전송/저장을 위해 형식만 바꾼다
       3. 해시 : 원본을 되돌릴 필요 없이 검증만 한다
  3. 실제 활용 예시

     1. 암호화

        1. 메신저 메시지 보호
        2. 파일 암호화
        3. HTTPS 통신
        4. 디스크 암호화
     2. 인코딩

        1. Base64로 이미지나 바이너리 데이터를 문자열로 표현
        2. URL에서 공백이나 특수문자 변환
        3. 문자 인코딩 UTF-8
        4. 이메일 첨부파일 전송 형식 변환
     3. 해시

        1. 파일 다운로드 후 무결성 확인
        2. 비밀번호 저장
        3. Git 커밋 식별
        4. 전자서명 과정에서 원문 대신 해시값 사용
  4. 이전 개념과의 연결

     1. 암호화 : 주로 기밀성과 연결
     2. 인코딩 : 데이터 표현 방식에 가까움
     3. 해시 : 무결성과 연결



* #### 해시의 기본 개념

  1. 개념

     * 입력 데이터를 고정된 길이의 값으로 바꾸어, 원본이 바뀌었는지 확인하는 일방향 변환
  2. 특징

     1. 고정 길이 출력
     2. 일방향성
     3. 결정성
     4. 민감한 변화
     5. 충돌 가능성
  3. 필요 이유

     * 원본 데이터를 직접 비교하지 않고도 데이터가 바뀌었는지 빠르게 확인하기 위해 필요
  4. 실제 활용 예시

     1. 파일 무결성 확인
     2. 비밀번호 저장
     3. 디지털 서명
     4. 데이터 식별
  5. 이전 개념과의 연결

     1. 해시 : 무결성과 연결
     2. 해시가 암호화·인코딩과 다르다는 점 학습



* #### 비밀번호를 복호화하지 않는 이유

  1. 개념

     * 비밀번호는 원문을 다시 알아낼 필요가 없기 때문에 복호화 가능한 형태로 저장하지 않고, 입력값을 다시 해시해서 비교
     1. 복호화 키 존재 -> DB와 복호화 키를 얻은 공격자가 비밀번호를 전부 원문으로 되돌릴 수 있음
  2. salt가 필요한 이유

     * salt : 각 비밀번호에 추가되는 고유한 랜덤 문자열
     1. 단순히 비밀번호만 해시 -> 같은 비밀번호를 쓰는 사람들은 같은 해시값
     2. salt를 붙이면 같은 비밀번호라도 결과 상이
  3. 실제 활용 예시

     1. 웹사이트 로그인
     2. 비밀번호 유출 피해 감소
     3. 같은 비밀번호 사용 여부 숨김
     4. 브루트포스 비용 증가
  4. 이전 개념과의 연결

     1. 암호화 : 원문을 나중에 다시 읽어야 할 때 사용
     2. 해시 : 원문을 다시 읽을 필요 없이 맞는지만 확인할 때 사용
     3. 비밀번호 : 원문을 다시 읽을 필요가 없으므로 해시로 저장
